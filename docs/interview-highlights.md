# 项目亮点摘要

这份文档用于面试时 1 到 2 分钟介绍项目，重点不是完整讲实现细节，而是帮助面试官快速理解项目价值、技术深度和工程判断。

## 1-2 分钟口述版

我做的是一个面向代码仓库的 AI Agent 项目，目标不是做一个简单的代码聊天页面，而是把 AI 能力真正落到研发工作流里。

这个系统支持导入本地仓库和 GitHub 仓库，先对代码做基础索引，再通过受控工具让 Agent 去检索代码、读取文件、定位符号，最后基于证据回答问题。回答不是只给自然语言结论，还会返回引用路径、行号和工具调用摘要，所以它更像一个“可解释的工程助手”，而不是黑盒问答。

在此基础上，我又把它往代码改动链路延伸了一步。系统可以生成 patch 草案和 unified diff 预览，支持单文件和多文件改动的确认式应用；应用前会校验文件哈希，避免覆盖已经变化的内容。改动之后还能自动运行白名单内的 lint/test，比如 `pytest` 和 `npm run typecheck`，形成一个最小可用的修改验证闭环。

技术上我用的是 `FastAPI + Next.js + SQLite + OpenAI Agents SDK`。我比较看重的不是“把模型接进来”，而是把系统设计成工具可控、证据可追踪、风险可约束、后续可扩展的结构。比如仓库来源统一抽象成工作区根目录，后面无论是索引、问答还是 patch/checks 都走同一条链路；又比如回答一定通过工具找证据，而不是直接把整个仓库扔进 prompt。

如果要总结这个项目的核心价值，我会说它体现的是：我不仅能把 LLM 接到产品里，还能把它设计成一个更接近真实工程协作方式的 Agent 系统。

## 面试时可以强调的 4 个点

- 不是普通聊天机器人，而是围绕代码任务设计的工具型 Agent。
- 回答基于工具检索和引用证据，可解释性比纯 prompt 问答更强。
- 不只做到“能问”，还做到了“能改”和“能验证”，形成了完整工作流。
- 在实现上保留了扩展性，后续可以继续接 embedding、rerank、AST 分析和更强的自动化。

## 如果面试官继续追问，可以展开这几件事

### 1. 为什么不用“直接把仓库丢给模型”

因为代码仓库天然有上下文大、信息噪声多、可追踪性差的问题。  
我采用的是工具访问方式，让 Agent 先用 `search_repo`、`read_file`、`find_symbol` 等工具逐步缩小范围，这样更节省上下文，也更容易解释模型为什么得出这个结论。

### 2. 这个项目最像真实工程的地方是什么

不是问答，而是 patch 和 checks 这一段。  
我没有让模型直接写文件，而是先出草案、先看 diff、再确认应用，并且在应用前做哈希校验，最后跑 lint/test。这一段更接近真实团队里的 review 和验证流程。

### 3. 你在这里体现了哪些工程判断

- 用 SQLite 先完成 MVP，而不是一开始就堆复杂基础设施
- 用白名单 checks 控制执行风险
- 用统一工作区抽象兼容本地仓库和 GitHub clone
- 用分阶段迭代的方式，把问答、改动、验证逐层串起来

### 4. 后续还能怎么演进

- 检索层：加入 embedding、rerank、AST 级符号能力
- 验证层：结合 diff 内容和历史失败记录推荐 checks
- 生产化：后台任务队列、运行审计、失败回滚和更细粒度权限控制

## 一句话版本

这是一个可调用工具、可引用证据、可生成改动建议并接入验证闭环的代码仓库 AI Agent，而不是一个只会解释代码的聊天界面。
